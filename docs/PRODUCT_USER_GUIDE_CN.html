为什么这套系统值得持续试用

从“开一张票”进化到“沉淀一套客户开票能力”

每一次真实使用，都不只是完成一次开票动作。系统会把客户、项目、赋码、执行结果和效率数据逐步沉淀下来，让后续开票更快、更稳、更容易复核。

档

客户档案建立

系统会围绕客户建立购买方、销售方、常用项目和历史明细档案。后续遇到同一客户或相似简称时，可以更快召回历史信息，减少重复录入。

简

客户简称识别

面对日常口头简称、平台简称或不完整客户名称，系统可以结合档案辅助识别，帮助您更快定位正确购买方和历史开票信息。

导

客户历史开票一键导入

对重复开票、周期性客户或常用项目，可一键导入客户历史开票信息，快速带出项目、税率、税收编码等内容，再由现场人员复核确认。

码

大模型智能赋码

本地规则优先，未命中时可对所有未命中明细一次性发起智能赋码。大模型只在官方候选池内推荐，并标记“需人工复核”。

库

内置手动赋码库

即使不使用大模型，也可以在草稿页手动查询官方税收分类编码，选中后批量填入税目大类、税收编码和税率。

数

端到端数据统计

系统记录材料识别、草稿生成、模板下载、税局上传、预览结果等过程，后续可统计开票数量、处理效率、失败原因和改进空间。

第 1 阶段减少重复录入把文字、截图、表格整理成草稿，统一复核后再进税局。
第 2 阶段形成客户档案沉淀购买方、简称、历史项目、常用税率和税收编码。
第 3 阶段优化赋码与复核本地规则、官方赋码库和大模型协同，降低查码压力。
持续使用后成为经营数据资产看到开票数量、效率、退回原因和客户结构变化。

操作步骤

  1. 确认“销售方 / 纳税主体”。
  2. 粘贴客户开票信息，或上传图片 / Excel / PDF。

  5. 进入草稿页后逐项复核并保存。
  6. 确认无误后下载税局 Excel 或发起上传税局。

04｜智能赋码

草稿先成型，赋码再二次处理

为了让您尽快看到可复核的草稿，首页“生成草稿”阶段不会因为税收编码大模型等待而卡住。系统先用本地规则快速生成草稿；本地未命中的明细，在草稿页统一处理。

本地命中

规则库能确定时直接填入分类和编码。

速度最快

未命中规则

显示“未命中规则”，不静默空白。

待处理

智能赋码

优先一次处理所有未命中行。

减少调用

人工复核

结果标记“智能推荐，需人工复核”。

不自动最终决定
页面状态建议操作说明
未命中规则 N 行点击“智能赋码未命中 N 行”多行合并成一次请求，通常比逐行点击更省成本、更少等待。
某一行失败点击该行“重试本行”只重试失败行，避免整批重复调用。
本地已命中但您觉得需要再确认点击“复核本行”用于定证检验，不是日常必点流程。
全部都想复核点击“智能复核全部明细”会增加等待和调用成本，只在必要时使用。
成本提示：多行未命中时，优先整体赋码。逐行调用会重复发送公共提示词、候选说明和格式要求，通常更贵也更碎片化。
05｜批量开票

多张截图 / 多个购买方集中处理

适用于餐饮平台截图、多个购买方、多张发票集中开具。批量与否只由首页“批量开具发票”勾选决定。

勾选批量开具发票

只有勾选后才进入批量模式。未勾选时，即使上传多个文件，也按单张发票材料处理。

上传多张截图

1-5 张图片默认按一张图一个业务单元，逐张识别；超过 5 张时回到更稳的混合模式。

进入批量开票工作表

在页面内复核 Sheet1 / Sheet2 / Sheet3，不要来回跳转。

批量明细填充

如果这一批都是同类项目，例如餐费，可在“批量明细填充”中确认项目、税收编码和税率，再一键导入所有明细行。

区域用途复核重点
Sheet 1：发票基本信息购买方、税号、金额、票种等重点核对购买方名称、税号、含税金额。
Sheet 2：发票明细信息项目、税率、税收编码、数量、单位等批量明细填充后仍需复核。
Sheet 3：待补全 / 异常项检查清单不是开票表；用于提醒哪些地方还要补。
图片建议：尽量一张图只保留一条完整开票卡片。拍屏、反光、裁切、多卡片混在一起会降低识别稳定性。
06｜功能总览

各项功能怎么理解

台

开票工作台

输入客户材料、选择单张或批量、识别当前税局主体、生成草稿。

稿

发票草稿

复核购买方、税号、金额、明细、税率、税收编码；保存修改不重新识别材料。

批

批量复核

批量截图或多张发票集中处理，支持 Sheet1/Sheet2/Sheet3 检查和下载税局 Excel。

档

主体与档案

查看销售方主体、购买方档案、历史常用项目，用于辅助推荐，不覆盖明确材料。

记

执行记录

查看识别方式、LLM 供应商、执行状态、现场操作留痕，方便排查。

退

税局退回处理

正常流程会自动记录税局退回信息；手工导入失败明细仅作为备用入口。

07｜现场更新

不用 GitHub，不手动覆盖文件

沈阳现场网络可能无法稳定访问 GitHub，因此现场机器不运行 git，也不要手动复制单个文件。后续只使用“最新累计更新包”。

📥收到更新包最新累计 zip
📁放到 updates重命名 latest.zip
🛡️自动备份APPLY_UPDATE
🔄覆盖更新写 VERSION / 日志
✅重启验证Ctrl + F5

标准更新流程

  1. 关闭当前工作台窗口 / Flask 黑色窗口。
  2. 把最新更新包复制到：
C:\invoice-assistant-staging-19d4851\updates\latest.zip
  1. 双击：
APPLY_UPDATE.bat
  1. 更新完成后重新启动：
02_START_INVOICE_ASSISTANT.bat

最后浏览器按 Ctrl + F5。

回滚流程

如果更新后明显异常：

  1. 关闭工作台窗口 / Flask 黑色窗口。
  2. 双击：
RESTORE_LAST_BACKUP.bat
  1. 重新启动工作台。
  2. 浏览器按 Ctrl + F5。

脚本会从 backups\LAST_BACKUP.txt 指向的最近备份恢复。

现场规则：不要运行 GitHub pull；不要手动复制单个文件；不要把私密配置打进更新包；每次只认一个最新累计包。
目录 / 文件作用
updates\latest.zip等待应用的最新更新包。
backups\每次更新前自动备份会被覆盖的文件。
update_logs\更新和回滚日志。
VERSION.txt当前已应用版本、备份位置和日志位置。
UPDATE_MANIFEST.txt更新包版本、文件清单和说明。
08｜常见问题

现场遇到问题先看这里

工作台网页打不开

确认已启动工作台黑色窗口，再手动打开：

http://127.0.0.1:5012

如果还是打不开，关闭黑色窗口后重新启动。

税局无法连接

确认已启动税局专用浏览器：

03_START_TAX_BROWSER.bat

并且浏览器窗口没有被关闭。

页面没有识别到主体

先在税局浏览器里完成登录，然后回工作台点击“识别当前税局主体 / 加载档案”。

批量截图数量不对

确认首页已勾选“批量开具发票”。未勾选时，系统会按单张发票材料处理。

智能赋码等待

多行未命中时，不要逐行点。优先点“智能赋码未命中 N 行”，一次处理。

税局返回失败

先看系统是否自动带回退回信息。若没有，再展开“税局退回处理”的备用入口手工导入失败明细。
